# StoryAI 개발계획서

> 유아 대상: 자녀 사진을 등록하면 AI가 그 아이를 주인공으로 하는 **동화책** 또는 **애니메이션 영상**을 만들어 주는 서비스.

---

## 1. 반영된 요구사항 (제품 스펙)

**입력**
- 고객이 자녀 사진 업로드
- 둘째/친구 등 **추가 인물 등록** → 추가된 인물까지 함께 주인공으로 스토리 생성
- 주제 선택: 공주 / 왕자 / 용사 등

**산출물 선택 (책 or 영상)**

| | 책 (Book) | 영상 (Video) |
|---|---|---|
| 스타일 | 동화형식 / 스케치형식 등 | 2D / 3D(베베핀풍) |
| 분량 | 24페이지 / 36페이지 | 약 2분 / 5분 |
| 미리보기 | (책 일부 페이지, 워터마크) | 약 20초 |
| 수령 방식 | PDF 다운로드 / 실물 책 배송 | 다운로드 |
| 결제 | 다운로드/배송 전 결제 | 다운로드 전 결제 |

## 2. 핵심 기술 난제 (성패를 가르는 부분 — 가장 먼저 검증)

### ⭐ (1) 캐릭터 일관성 (Character Consistency) — 최우선 리스크
사진 속 아이가 **24~36페이지 / 2~5분 영상 전반에 걸쳐 같은 캐릭터로 유지**되어야 함. 이게 안 되면 제품 자체가 성립 안 됨.
- 권장 접근: 사진 → **캐릭터 시트(정면/측면/표정 세트)를 먼저 1회 생성**해 "레퍼런스"로 고정 → 이후 모든 페이지/장면은 이 레퍼런스를 참조해 생성 (reference-image / IP-Adapter 계열, 또는 캐릭터별 경량 파인튜닝).
- **스타일라이즈 권장**: 실사 얼굴을 그대로 쓰기보다 동화/애니 캐릭터로 변환. 이유 (a) 페이지 간 일관성이 훨씬 잘 잡힘 (b) 유아 초상 실사 노출을 피해 **개인정보·안전 측면에서 바람직**.

### (2) 2D vs 3D 영상
- **2D**: 씬별 일러스트(캐릭터 일관) → 이미지-투-비디오 모션 / 카메라 워크. 상대적으로 저렴·빠름.
- **3D(베베핀풍)**: 실제 3D 리깅·렌더는 비용·난이도가 급상승. 초기엔 **"3D 렌더 스타일 이미지 + 모션"으로 근사**, 진짜 파이프라인은 후순위.

### (3) 비용·시간
- 5분 영상 / 36p 삽화는 AI 생성 비용과 시간이 큼 → **비동기 잡 큐 + 크레딧/가격 산정 + 미리보기 게이팅**이 구조적으로 필요 (지금 워크플로우 엔진 기반 확장).

### (4) 안전·법무 (유아 대상 필수 고려)
- 아동 사진 업로드 → 개인정보 처리방침·보관/삭제 정책, 보호자 동의(결제 주체=보호자).
- 스토리/이미지 안전 필터(연령 적합), 초상권·저작권.

## 3. 아키텍처 / 도메인 확장

현재는 `VideoJob` 단일 모델. 아래로 **일반화** 필요:

```
Project (주문·생성 단위)
  ├─ type: BOOK | VIDEO
  ├─ theme: PRINCESS | PRINCE | HERO | ...
  ├─ style: (BOOK) STORYBOOK | SKETCH ... / (VIDEO) ANIM_2D | ANIM_3D
  ├─ spec:  (BOOK) pages 24|36        / (VIDEO) durationSec 120|300
  ├─ status: 상태 머신 (기존 엔진 재사용)
  └─ characters: [Character...]

Character (주인공/추가 인물)
  ├─ role: CHILD | SIBLING | FRIEND
  ├─ name, sourcePhoto
  └─ characterSheet: [참조 이미지들]  ← 일관성의 핵심

Asset (산출물 조각)
  └─ pageImage | audioNarration | videoSegment | preview | finalPdf | finalVideo

Order / Payment
  ├─ product: PDF_DOWNLOAD | PRINT_BOOK | VIDEO_DOWNLOAD
  ├─ preview gating (미리보기까지 무료, 전체는 결제 후)
  └─ payment 연동

Fulfillment (실물 책)
  └─ 인쇄·배송 벤더 연동 / 배송 상태
```

### 파이프라인 분기
- **BOOK**: 스토리(N페이지 구성) → 페이지별 삽화(캐릭터 일관) → 텍스트 조판 → PDF 생성 → 미리보기(일부 페이지+워터마크) → **결제** → 전체 PDF / 인쇄 발주
- **VIDEO**: 스토리 → 씬 분리 → 키프레임(캐릭터 일관) → 모션/영상화 → 나레이션(TTS) → 자막 → FFmpeg 합성 → 20초 미리보기 → **결제** → 풀영상

## 5. 단계별 로드맵 (MVP 우선 · "가장 먼저 눈에 보이는 결과")

- **Phase 0 — 캐릭터 일관성 스파이크** ⭐ *(코드보다 먼저)*
  사진 1장 → 캐릭터 시트 → 서로 다른 3장면에서 **동일 캐릭터** 재현되는지 PoC. 여기서 이미지 벤더·방식 확정.
- **Phase 1 — 책 MVP**
  단일 주인공 · 동화형식 · 24p · 스토리(Claude)+삽화 생성 · PDF 다운(무료/테스트) · 웹 업로드 UI · 진행상황 표시.
- **Phase 2 — 결제 + 미리보기 게이팅 + 스케치 스타일 + 36p**
- **Phase 3 — 다중 캐릭터** (둘째/친구 추가, 이들 중심 스토리)
- **Phase 4 — 영상 MVP** (2D · 2분 · 20초 미리보기 · 결제)
- **Phase 5 — 영상 5분 + 3D풍 스타일**
- **Phase 6 — 실물 책 배송 · 회원/주문내역 · 관리자 · 운영**

> **왜 책 먼저?** 영상 대비 생성비용·시간이 훨씬 작고, 핵심 난제(캐릭터 일관성)를 가장 빠르고 싸게 검증 가능. 여기서 잡은 캐릭터 파이프라인을 영상에 그대로 재사용.

## 7. 확정된 결정사항 (2026-07-10)
- ✅ **책 + 영상 둘 다** 제공 — 고객이 프로젝트마다 선택.
- ✅ **결제는 나중에** — 지금은 생성 파이프라인 + 다운로드까지. 미리보기/게이팅 구조는 잡되 실제 PG 연동은 후순위.
- ✅ **실물 책**: 인쇄 벤더 API 연동 안 함. 고객이 "실물 요청" 하면 그 **요청 정보(주문+배송)만 수집** → 운영자가 수기로 PDF를 인쇄소에 발주·배송. 즉 `physicalBookRequested` 플래그 + 배송정보만 저장.
- ✅ **캐릭터: 실사 느낌 유지** (스타일라이즈 아님).
  - ⚠️ 함의: (1) 페이지/영상 간 얼굴 일관성 확보가 더 어려움 → **레퍼런스/FaceID 기반** 파이프라인 필수. (2) 유아 실사 얼굴을 다루므로 **보호자 동의·이미지 암호화 보관·자동 삭제(TTL) 정책**을 초기부터 반영.
